Template Setup Guide

1 Purpose

This hidden page explains how to initialize a new living-document repository from this template and configure GitHub + Zenodo correctly.

2 Setup Flow

3 Repository Prerequisites

  1. Enable GitHub Actions in the new repository.
  2. Ensure Pages deployment from gh-pages is allowed.
  3. Keep these workflows present:
    • .github/workflows/pr-check.yml
    • .github/workflows/release.yml
  4. Keep Quarto project render list including:
    • IPBES_Data_Vision.qmd
    • older-releases.qmd
    • RELEASE.qmd
    • setup.qmd
  5. Keep shared release runner script present:
    • scripts/release_pipeline.py

4 Zenodo Setup (Initial Bootstrap)

This template workflow creates new versions from an existing Zenodo concept record.
Therefore, each environment needs one initial manual Zenodo publication first.

Important:

  1. Zenodo sandbox and production are separate systems.
  2. You must bootstrap each one independently.

Zenodo endpoints:

  1. Sandbox UI: https://sandbox.zenodo.org
  2. Sandbox API base: https://sandbox.zenodo.org
  3. Production UI: https://zenodo.org
  4. Production API base: https://zenodo.org

Initial bootstrap steps (run first in sandbox, then production):

  1. Create Zenodo access token with deposition permissions.
  2. Create a new upload in Zenodo manually.
  3. Fill required metadata.
  4. Upload an initial PDF file.
  5. Publish the record manually in Zenodo.
  6. Copy the concept DOI (“all versions DOI”) from the published record page.
  7. Set concept_doi in IPBES_Data_Vision.qmd to that concept DOI.

After bootstrap:

  1. Release workflow can derive ZENODO_CONCEPTRECID from concept_doi.
  2. Release workflow can reserve new version DOIs automatically.
  3. Workflow uploads to Zenodo draft, but final draft publish/mint remains manual.

5 Required Frontmatter

Edit IPBES_Data_Vision.qmd and set:

  1. title
  2. version
  3. release_date
  4. concept_doi
  5. release_notes
  6. editors
  7. contact_email

Notes:

  1. version is used for release PDF naming.
  2. concept_doi is used to derive ZENODO_CONCEPTRECID in release workflow.
  3. concept_doi must come from the initial manually published Zenodo record.
  4. Version DOI is injected automatically at release time (IPBES_DOI).

6 GitHub Variables and Secrets

Set these at repository level in GitHub:

  1. Open the repository on GitHub.
  2. Go to Settings -> Secrets and variables -> Actions.
  3. In Variables, add:
    • ZENODO_TARGET
    • ZENODO_API_BASE
    • ZENODO_SANDBOX_API_BASE
  4. In Secrets, add:
    • ZENODO_TOKEN
    • ZENODO_SANDBOX_TOKEN

Keep ZENODO_TARGET as the single environment switch (sandbox or production).

6.1 Variables

  1. ZENODO_TARGET
    • allowed values: sandbox or production
    • if unset/invalid, workflow defaults to production
  2. ZENODO_API_BASE
    • production API base, e.g. https://zenodo.org
  3. ZENODO_SANDBOX_API_BASE
    • sandbox API base, e.g. https://sandbox.zenodo.org

6.2 Secrets

  1. ZENODO_TOKEN
    • production Zenodo token
  2. ZENODO_SANDBOX_TOKEN
    • sandbox Zenodo token

7 Environment Selection Logic

The release workflow resolves active Zenodo settings once, exports:

  1. ACTIVE_ZENODO_TARGET
  2. ACTIVE_ZENODO_API_BASE
  3. ACTIVE_ZENODO_TOKEN

8 Recommended Rollout

  1. Set ZENODO_TARGET=sandbox.
  2. Run make release-local once to validate token + concept ownership before CI release.
  3. Publish a test GitHub release.
  4. Confirm in run logs:
    • target resolution succeeded
    • DOI reserved in sandbox
    • website and PDF rendered with reserved DOI
    • PDF + release notes uploaded to Zenodo draft
  5. Confirm _site deployed correctly and release assets exist.
  6. Switch ZENODO_TARGET=production.
  7. Publish official release.
  8. Manually publish/mint Zenodo draft in Zenodo UI.

9 What Is Automatic vs Manual

Automatic in release workflow:

  1. Reserve Zenodo version DOI.
  2. Append DOI line to GitHub release notes.
  3. Build HTML/PDF with reserved DOI.
  4. Upload PDF + notes to Zenodo draft.
  5. Upload PDF to GitHub release.
  6. Deploy _site to gh-pages.

Automatic in local release runner (make release-local):

  1. Resolve active Zenodo target/token/base.
  2. Read frontmatter metadata.
  3. Reserve Zenodo DOI.
  4. Build website/PDF with reserved DOI.
  5. Upload PDF + notes to Zenodo draft and verify submitted=false.

Manual:

  1. Publish/mint Zenodo draft record.

10 Validation Checklist for New Template Repos

  1. make build succeeds locally.
  2. PR checks are green on a test PR.
  3. Sandbox release run is green.
  4. _site/index.html, _site/older-releases.html, and latest PDF exist.
  5. release.html and setup.html render but are not linked in site navigation.
  6. Production release run is green.
