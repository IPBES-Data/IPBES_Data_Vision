Release Process

1 Purpose

This page explains the release lifecycle for the IPBES Data Management Vision:

  • what runs on pull requests,
  • what runs when a GitHub release is published,
  • how GitHub Actions interacts with Zenodo,
  • and where manual Zenodo publish/minting begins.

Core implementation note:

  • Release orchestration is centralized in scripts/release_pipeline.py.
  • The same script powers both local release runs and GitHub release workflow runs.

5 Stage Matrix

Stage Trigger Automation Output
PR check Pull request pr-check.yml Validation status + artifacts
Target selection Release publish release.yml Active Zenodo base/token
DOI reservation Release publish release.yml + Zenodo Reserved version DOI
Build Release publish release.yml HTML + versioned PDF
Zenodo draft upload Release publish release.yml + Zenodo Draft updated (PDF + notes)
GitHub release upload Release publish release.yml PDF asset on release
Deploy Release publish release.yml Updated _site on gh-pages
Final publish/mint Manual Zenodo UI Minted/published record

6 Local Release Runner

Use:

  1. make release-local

Behavior:

  1. Runs shared pipeline subcommands in order:
    • resolve-target
    • read-frontmatter
    • reserve-doi
    • build-artifacts
    • upload-zenodo-draft
  2. Reserves DOI and updates Zenodo draft.
  3. Does not mint/publish Zenodo draft.
  4. Does not update GitHub release notes or deploy gh-pages.

State/output contract (.release-state.env):

  1. IPBES_DOI and IPBES_DOI_URL
  2. ZENODO_DRAFT_ID
  3. PDF_NAME, PDF_PATH, and LATEST_PDF

7 Configuration

Repository variables and secrets:

  1. ZENODO_TARGET: sandbox or production (defaults to production if unset/invalid)
  2. ZENODO_API_BASE: production Zenodo API base
  3. ZENODO_SANDBOX_API_BASE: sandbox Zenodo API base
  4. ZENODO_TOKEN: production token (used when ZENODO_TARGET=production)
  5. ZENODO_SANDBOX_TOKEN: sandbox token (used when ZENODO_TARGET=sandbox)

Switching environment is done by changing only ZENODO_TARGET.

8 Frontmatter Inputs

The release workflow reads these fields from IPBES_Data_Vision.qmd frontmatter:

  1. version
  2. concept_doi
  3. release_date
  4. release_notes
  5. editors
  6. contact_email

Behavior rules:

  1. Reserved DOI is injected as IPBES_DOI at release build time.
  2. If DOI is unavailable, output displays **Will be inserted upon release**.
  3. concept_doi is used for latest-version reference and all-releases lookup.
  4. ZENODO_CONCEPTRECID is derived from concept_doi.

9 Failure Conditions

Release workflow fails on:

  1. Missing/invalid version or concept_doi.
  2. Missing selected base/token for active ZENODO_TARGET.
  3. Zenodo DOI reservation failure (including no candidates, unauthorized token, or invalid candidate ids).
  4. Build failure.
  5. Zenodo draft upload failure.
  6. GitHub release asset upload failure.
  7. gh-pages deployment failure.

10 Post-Release Checklist

  1. Workflow run is green.
  2. Site is updated on gh-pages.
  3. GitHub release has attached PDF.
  4. HTML/PDF show reserved DOI.
  5. Zenodo draft contains uploaded PDF and release notes.
  6. Release notes include exactly one DOI line.
  7. Zenodo draft is published manually when ready.
